# EXPERIMENTS.md — Experiment Strategy & Status

Living document tracking the overarching experimental plan and where each piece stands.
Companion to [CLAUDE.md](CLAUDE.md) (design/spec) and [hpc/experiment_setup.md](hpc/experiment_setup.md)
(compute budget). When in doubt about *what we are running and why*, this file is authoritative.

## 0. Overall Goal

Show, **through experiments, which RL approach is most suitable** for this infrastructure-maintenance
MDP. Strategy: **start small, then scale up**, across three experiments:

| Experiment | Instance | Scope | Replication |
|---|---|---|---|
| **Exp 0 — Exploration** | `instance_10p` (N=10) | Wide screen of *all* algorithm variants; cheap, just to see what works | **1 per config** |
| **Exp 1 — Wide selection, larger instance** | ~N=30 (likely Sioux Falls, ~30 projects) | The subset of approaches that worked in Exp 0, run for real | 5 seeds (stats) |
| **Exp 2 — Scale-up case study** | Large case study (TBD) | Only the best algorithms from Exp 1 | 5 seeds (stats) |

Exp 0 is exploratory: a single replication per config is enough to spot which variants are viable and
prune the grid. Statistical claims (rliable bootstrap CIs) come from Exp 1 onward, where replication
matters. Each step is small enough to be tractable before committing compute to the next.

## Experiment 0 — Exploration on `instance_10p`

> **⚠️ Instance regenerated 2026-06-19 — exp0 re-run in progress.** A predictability analysis
> ([docs/i10p_predictability_analysis.md](docs/i10p_predictability_analysis.md)) found the original
> `instance_10p` was **noise-dominated**: failure timing was essentially unpredictable (time-to-failure
> CV ≈ 0.6; current state explained only R²≈0.58 of cost-to-go), which *structurally caps* anticipatory
> RL — explaining why MC rollout barely beat reactive on the mean. `instance_10p.json` was regenerated
> to be **anticipatable** (`alpha0_mean` 0.05→0.8 → TTF-CV ≈ 0.22, same 60-yr lifetime); the old one is
> saved as `instances/instance_10p_noisy_alpha0p05.json`. **The first exp0 results (on the noisy
> instance) are archived at `results/exp0_noisy_alpha0p05/` — they are the "unpredictable" arm and a
> finding in their own right** (notably: MC rollout cut outcome-cost **CV 0.74→0.23** — its value was
> *risk reduction*, not mean anticipation → report CVaR/P90, not just mean). Exp 0 is now being re-run
> on the predictable instance.

All Exp 0 runs use `instances/instance_10p.json`, Sioux Falls network, `fast` TAP backend.
Evaluation uses shared CRN across agents (paired comparison), so costs are directly comparable.
**One replication per config** — exploratory only, no aggregate statistics.

> **0A → 0B is a pipeline, not two parallel batches.** The Optuna heuristics (0A) must run **first**:
> their best-found parameters are the heuristic that the 0B agents consume (ADP warmstart policy,
> MC-rollout base policy). Submit 0A, read each run's `best_params.json`, inject those values into the
> relevant 0B configs, *then* submit 0B. See [Gaps](#gaps--open-items) for the current wiring status.

### 0a. Optuna-tuned heuristics (baselines **and** input to 0B)

Tuned classical policies that (1) set the baseline the RL methods must beat and (2) **supply the
heuristic that the 0B agents are warm-started / rolled out from**. **Single-threaded** (see Compute /
conventions). Each run writes `results/<run>/best_params.json`.

> ⚠️ The Optuna `best_value` is **optimistically biased** (~1.8x) vs. held-out cost — it is the min
> over many noisy trials (winner's curse), so **always report the held-out `"evaluation"` metric**, not
> `best_value`. To mitigate, tuning now uses `WilcoxonPruner` + CRN across trials and **100 tuning
> episodes** (was 30); see [hpc/registry_conventions.md](hpc/registry_conventions.md).

| Config | Heuristic |
|---|---|
| [i10p_optuna_reactive.json](configs/i10p_optuna_reactive.json) | reactive (threshold) |
| [i10p_optuna_paced.json](configs/i10p_optuna_paced.json) | paced |
| [i10p_optuna_perasset.json](configs/i10p_optuna_perasset.json) | reactive-per-asset |

### 0b. Heuristic-fed / learning algorithms

These either warm-start from, bootstrap off, or roll out a heuristic policy — **the heuristic and its
parameters should come from 0A's tuned output** (currently placeholders, see Gaps).

**(i) ADP + variants — full grid.** Dimensions:
- **Action generation:** normal (local search) / sequential
- **Initial policy:** empty / heuristic warmstart
- **Buffer management:** FIFO / worst-errors (`lowest_error`) / stochastic knockout
- **VFA:** XGBoost / neural net

Full grid = 2 × 2 × 3 × 2 = **24 configs**, named `i10p_adp_{normal,seq}_{empty,policy}_{fifo,lowesterror,knockout}_{xgb,nn}.json`.
All 24 are generated by [configs/gen_phase1_adp_grid.py](configs/gen_phase1_adp_grid.py) (re-run to
regenerate). `policy` variants warm-start from a reactive heuristic (thr=0.95, 1667 warmstart eps).

**(ii) MC rollout** — [i10p_rollout_*.json](configs/), with `empty` vs `policy` initial action, and
a sequential variant (`seq_rollout`).

**(iii) PPO with curriculum learning** — curriculum-based policy-gradient,
[i10p_ppo_curriculum.json](configs/i10p_ppo_curriculum.json).

### 0b-ii. MC rollout — 4/4 built · ⚠️ first run was buggy → FIXED, needs re-run
First 0B run produced **pathological ~10B eval (99% risk, do-nothing)**. Root cause (found + fixed):
a **vanishing rollout horizon** — configs set `max_steps` but `configs.py` read `rollout_horizon`, so
the window was dropped and `_estimate_q` used `max_steps = cfg.T - t_next`, which goes ≤0 across the
eval tail (T+tail) → zero rollouts → do-nothing → escalating C_risk. Fixes in `agents/rollout.py`
(horizon now a fixed t-independent window), `configs.py` (`_check_extra_keys` rejects unknown extras),
and the 4 configs (`max_steps`→`rollout_horizon: 100`). Local CRN validation: 3.9e9/92%-risk →
~5.4e8/~6%-risk (vs reactive base 3.6e8 at shortened T). Uses **adaptive sequential-Wilcoxon**
budgeting (`rollout_selection: adaptive`, `p_threshold 0.02`, `min/max_rollouts 20/100`).
**Open:** full 50-ep Snellius eval not yet run — re-run to confirm it lands near/below 830M base; may
only *match* base at this budget (consider higher `n_rollouts` / less-degenerate base ~0.7 to beat it).

## Compute Budget (from [hpc/experiment_setup.md](hpc/experiment_setup.md))

- **Replication:** Exp 0 = **1 per config** (exploration). Exp 1 / Exp 2 = **5 seeds** per config
  (report with `rliable`: stratified bootstrap CIs). **Optuna tuning is never replicated** (1 run).
  See [hpc/registry_conventions.md](hpc/registry_conventions.md).
- **Workers:** learner configs use `n_workers = 16` with `--cpus-per-task=16` (the Snellius minimum
  shareable slot, so it's the smallest billed unit anyway — see
  [hpc/registry_conventions.md](hpc/registry_conventions.md); no `--mem-per-cpu`). **Optuna heuristic
  search is single-threaded** (`n_workers = 1`) — sequential, ignores extra workers. Use `n_workers:
  1` everywhere on the laptop.
- **Walltime:** 24 h per run (`time_budget: 86400`).
- Runs dispatched via HyperQueue; `hpc/run_task.py` applies the per-entry `seed`/`run_name` overrides
  from the registry.
